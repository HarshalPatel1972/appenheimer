# Contributing

> Appenheimer values thoughtful contributions over fast contributions.

---

# Purpose

This document explains how contributors should work on Appenheimer.

Whether human or AI-assisted, every contribution should improve the project without compromising its philosophy, architecture, or maintainability.

---

# Before Contributing

Every contributor should read the following documents before making changes:

1. 00-vision.md
2. 01-engineering-principles.md
3. 10-non-goals.md
4. Relevant RFCs
5. Relevant ADRs

Understanding the project is required before changing it.

---

# Development Workflow

Every feature follows the same workflow.

Idea

↓

Documentation / RFC

↓

Implementation Plan

↓

Architecture Review

↓

Implementation

↓

Code Review

↓

Merge

No implementation should skip this process.

---

# Pull Request Guidelines

Every pull request should:

- Solve one problem.
- Remain focused.
- Be easy to review.
- Include clear reasoning.
- Follow the engineering principles.

Avoid mixing unrelated changes.

---

# Code Standards

Every contribution should:

- Follow the project's engineering principles.
- Prefer clarity over cleverness.
- Keep functions small.
- Avoid unnecessary abstractions.
- Maintain consistency with the existing codebase.

---

# Documentation

Documentation is part of the product.

Whenever behavior, architecture, or philosophy changes:

- Update the relevant document.
- Record significant decisions in the Decision Log.
- Keep documentation synchronized with implementation.

---

# AI-Assisted Development

AI is treated as an engineering partner, not an autonomous decision maker.

AI may:

- Implement approved designs.
- Refactor code.
- Generate tests.
- Suggest improvements.

AI must not:

- Invent product features.
- Change project philosophy.
- Override documented decisions.
- Introduce unnecessary complexity.

---

# Code Review

Every review should evaluate:

- Correctness
- Simplicity
- Maintainability
- Readability
- Performance
- Consistency
- Testability

Working code alone is not sufficient.

---

# Communication

When proposing significant changes:

- Explain the problem.
- Explain the proposed solution.
- Explain the trade-offs.

Focus discussions on reasoning rather than personal preference.

---

# Final Principle

> Contribute in a way that makes the project easier to understand than it was before your change.
